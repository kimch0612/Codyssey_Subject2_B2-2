실전 Git 협업 워크플로우
----------

### 과제 수행 체크리스트
- [ ] 3~5인 팀 구성
- [ ] GitHub 팀 저장소 생성
  - [ ] 옵션 A: Organization 저장소
  - [ ] 옵션 B: 개인 저장소 + Collaborator 초대
- [ ] 기본 구조 생성
  - [ ] `README.md`
  - [ ] `SUBMISSION.md`
  - [ ] `docs/`
  - [ ] `src/` 또는 결과물 폴더
  - [ ] 필요 시 `team/`
- [ ] `main` 브랜치 보호 설정
  - [ ] 직접 push 금지
  - [ ] PR로만 병합
  - [ ] 최소 1명 approve 필요

**협업 규칙**

- [ ] GitHub Flow 적용
  - [ ] `main`은 항상 안정 상태
  - [ ] 작업은 `feature/*` 브랜치에서 진행
- [ ] 브랜치 네이밍 규칙 정하기
- [ ] 커밋 메시지 컨벤션 정하기
  - [ ] 예: `feat: add string utils`
  - [ ] 금지: `update`, `fix`, `temp`, `wip`, `final`
- [ ] PR 작성 규칙 정하기
  - [ ] What
  - [ ] Why
  - [ ] How
  - [ ] `Closes #이슈번호`
- [ ] 코드 리뷰 규칙 정하기
  - [ ] “LGTM”만 쓰지 않기
  - [ ] 파일/라인/로직 근거가 있는 코멘트 남기기

**Issue / PR 작업**

- [ ] 모든 작업을 Issue로 만들기
- [ ] 각 PR에 연결 Issue 적기
  - [ ] `Closes #1`
  - [ ] `Fixes #2`
- [ ] 팀원별 최소 PR 2개 생성 및 병합
- [ ] 팀원별 최소 리뷰 2개 작성
- [ ] 팀원별 본인 PR에서 리뷰 반영 경험 최소 1회 남기기
  - [ ] 수정 커밋
  - [ ] 답글
  - [ ] 반영 완료 코멘트

**필수 문서**

- [ ] `docs/CONTRIBUTING.md`
  - [ ] GitHub Flow 설명
  - [ ] 브랜치 네이밍 규칙
  - [ ] 커밋 메시지 규칙
  - [ ] PR 규칙
  - [ ] 코드 리뷰 규칙
  - [ ] 충돌 발생 시 대응 흐름
- [ ] `docs/conflict-resolution.md`
  - [ ] 충돌 기록 최소 2회
  - [ ] 그중 1회는 비자명 충돌
  - [ ] 상황, 충돌 내용, 해결 과정, 결과, 배운 점 기록
- [ ] `docs/troubleshooting-log.md`
  - [ ] `commit --amend`
  - [ ] `reset --soft HEAD~1`
  - [ ] `revert`
  - [ ] `stash / stash pop`
  - [ ] 각 팀원 최소 1개 시나리오 기록 참여
- [ ] `SUBMISSION.md`
  - [ ] 저장소 URL
  - [ ] 팀원 목록
  - [ ] 팀원별 Issue 링크
  - [ ] 팀원별 PR 링크
  - [ ] 주요 문서 링크
  - [ ] 충돌/트러블슈팅 증빙 링크
  - [ ] git log 증빙 링크 또는 텍스트 위치

**충돌 실습**

- [ ] 충돌 상황 의도적으로 만들기
- [ ] 충돌 해결 최소 2회 기록
- [ ] 비자명 충돌 최소 1회 만들기
  - [ ] 같은 파일의 같은 근처 라인 수정
  - [ ] 또는 파일 이동/삭제 vs 내용 수정
- [ ] 충돌 마커 의미 설명 가능하게 정리
  - [ ] `<<<<<<<`
  - [ ] `=======`
  - [ ] `>>>>>>>`

**간단한 결과물 택 1**

- [ ] A안: 유틸 함수 모음
  - [ ] 팀원별 함수 1개 이상
  - [ ] 간단 사용 예시
- [ ] B안: 팀 소개 문서
  - [ ] `team/`에 팀원별 소개 파일
  - [ ] `README.md`에서 링크 제공
- [ ] C안: 학습 노트
  - [ ] 팀원별 마크다운 노트 1개 이상
  - [ ] `README.md`에서 목차 제공
- [ ] 선택한 결과물에 대해 팀원별 최소 1개 기여 커밋 남기기

**Git 히스토리 증빙**

- [ ] PR merge 기록이 보이도록 히스토리 남기기
- [ ] `git log --oneline --graph --all` 결과 저장
- [ ] 텍스트 또는 스크린샷으로 제출물에 연결

**보너스**

- [ ] `git rebase -i`로 개인 feature 브랜치 히스토리 정리
  - [ ] squash
  - [ ] reword
  - [ ] 정리 전/후 히스토리 비교 문서화
- [ ] `.github/CODEOWNERS` 추가
  - [ ] 파일/폴더별 책임 리뷰어 지정
  - [ ] 자동 리뷰어 요청 흐름 확인
